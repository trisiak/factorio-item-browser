# Data plane: self-hosted pack data for FIB (and later FBE)

The plan for making this app independent of FactorioLab by hosting our own
pack data, generated by the fbe fork's exporter — published first from fbe's
existing Pages data dir, later from a dedicated data repo. This document is
the durable design record and progress tracker for that work, spanning three
repos:

- **this repo** (`trisiak/factorio-item-browser`) — gains a second data-source
  adapter behind the existing provider seam;
- **`trisiak/factorio-blueprint-editor`** (fbe) — its Rust exporter
  (`packages/exporter/`) grows a browser-oriented output artifact, and its
  existing Pages data dir hosts that artifact in slice 1;
- **the data repo** (new, slice 2 — name TBD, working name
  `trisiak/factorio-pack-data`) — a dedicated GitHub Pages site the pack
  artifacts eventually move to, created when the big `editor/` tiers move.

Keep this doc reconciled with the work in the same change that lands it, like
`static-fork.md`. The fbe repo's issue tracker is the index on its side
(issue #8 = data hosting; issue #29's pause note = the licensing thread);
cross-link when slices land there.

## Goals

1. **Independence from FactorioLab.** Our packs come from our own pipeline;
   FactorioLab remains an *allowed* source kind in the manifest (it's how kr2 /
   pys / other ready-made packs stay one entry away), but nothing we ship
   depends on their publishing cadence or schema stability.
2. **Data FactorioLab structurally lacks:** descriptions, real research-unit
   counts, machine slot counts, exact mod sets, and (later) multiple locales.
3. **Space Exploration on Factorio 2.0.** The fbe fork already generates
   SE 0.7.56 on Factorio 2.0.76 (its `space-exploration` pack) — the same data
   basis serves this app, closing the sxp version-basis gap
   (`static-fork.md`, quirk #6).
4. **Carry the planned data-flow improvements of both projects:** fbe #8
   (evict ~434 MB of pack data from the fbe repo), fbe #50 phase 5c (DOM icons
   for the blueprint library — free once the browser icon sheet exists), and
   the fbe #29 licensing trajectory (see "The atlas trajectory" below).

## The one design call that shapes everything

**Shared hosting and provenance, per-consumer schemas.** The editor and the
browser need different data (raw prototype slices + GPU atlases vs. a compact
catalog + a CSS spritesheet); merging them into one shape would be wrong. So
one exporter run per pack emits **multiple artifacts**, each shaped for its
consumer, published side by side:

```
<data-host>/
  packs.json                 shared manifest (see below)
  <pack-id>/
    browser/                 ← this app (and fbe's DOM UI); small, slice 1
      catalog.json           items/fluids/recipes/technologies/machines
      icons.webp             64 px cells, 2 px gutter (FactorioLab-compatible geometry)
      icons.json             icon-id → sheet rect (+ optional iconText overlays)
    editor/                  ← fbe only; large, licensing-heavy, slice 2
      data.json              the curated data.raw slice fbe renders from
      __<mod>__/*.basis      GPU-compressed sprite atlas
```

The `browser/` tier is a few MB per pack; the `editor/` tier is 68–209 MB per
pack. Keeping them separable is what lets slice 1 ship without moving the big
data, and what keeps the fbe #29 trajectory alive: the full-quality atlas is
one deletable/replaceable artifact, not the foundation.

**The host is a base URL, staged in two steps.** In slice 1 `<data-host>` is
fbe's existing published data dir —
`https://trisiak.github.io/factorio-blueprint-editor/data/` — which already
serves `<pack-id>/…` with CORS and already carries the manifest; the
`browser/` tiers are simply committed alongside the editor data fbe ships
today. The dedicated data repo is deferred to slice 2, where it genuinely
earns its existence (the ~434 MB editor tiers move, and flat regeneration
history starts mattering). This creates no lock-in: this app couples only to
a published base URL behind its source adapter (per CLAUDE.md), so the later
move is a per-pack URL swap in `packs.ts` — combination ids don't change, so
user storage scoping and shareable URLs are untouched. Only coordination
needed then: keep the old URLs serving until this app's redeploy.

**Manifest.** The manifest *is* the fbe exporter's existing
`data/output/packs.json` (already published at the host root), extended
additively per pack: `generated` (timestamp, doubles as a cache-buster),
`schemaVersion` (of the browser catalog), and `artifacts` (which tiers
exist). Additive is safe on both ends — the exporter's `Pack` deserializer
ignores unknown fields, and the fbe editor reads only `id`/`label`/`mods`.

GitHub Pages serves `Access-Control-Allow-Origin: *`, so both apps fetch
cross-origin without any proxy. Pages soft-caps a site at ~1 GB and
100 GB/month bandwidth — comfortable for browser tiers; re-evaluated in
slice 2 when the editor tiers move (fallback if ever needed: editor tiers as
release assets behind a CDN, invisible to consumers because the manifest
carries the URLs).

## Prior art / pipeline decisions

Studied before committing to this design (2026-07-24):

- **Upstream FIB** (`factorio-item-browser` org): a PHP microservice fleet —
  Combination API, export workers launching Factorio with a dump mod, an icon
  renderer they eventually **rewrote in Go** for performance, FTP upload,
  DB import. Built for *arbitrary user-requested mod combinations on demand* —
  the exact problem the static fork deleted. Nothing to reuse
  implementation-wise (GPL PHP, service-shaped), but two lessons: the rich
  data model (descriptions, machines, per-language translations) this app's
  `transfer.ts` still carries is what their pipeline delivered — populating it
  from our exporter restores original capability; and icon composition was
  painful enough to force a rewrite — don't reimplement it.
- **FactorioLab**: the `factoriolab-export` *mod* (MIT) composes icons
  in-game via `LuaRendering.draw_sprite`, with documented scaling glitches.
  But their **maintainer pipeline** (`scripts/` in `factoriolab/factoriolab`)
  runs Factorio with the built-in CLI flags
  **`--dump-data --dump-prototype-locale --dump-icon-sprites`** and processes
  the dumps in TypeScript (Spritesmith/Sharp → `icons.webp`, 64 px cells,
  2 px gutters; per-language i18n files). The critical flag is
  `--dump-icon-sprites`: **the game itself renders every prototype's fully
  composed icon** (layers, tints, overlays, correct scaling) to PNGs.
- **The fbe Rust exporter** (~1,000 lines Rust + ~580 Lua): reviewed in full;
  in good shape and worth continuing — loaded-mod-set verification before the
  long atlas build, stale-dump deletion, load-order-correct dumping via
  generated optional dependencies, deterministic locale override order,
  `.part` streaming downloads with bad-credential detection, zip-slip guards,
  incremental mtime/size atlas cache with pruning. Its one real weak spot is
  the hand-rolled `.cfg` locale resolver (English-only, top-level locale only,
  naive template substitution).

**Decision:** the `browser/` artifact is built from **Factorio's own dump
flags**, not by extending the injected mod's reach: `--dump-icon-sprites` for
composed icons (we never write icon-composition logic), and
`--dump-prototype-locale` for engine-resolved names *and descriptions*
(strictly better than the `.cfg` parser; also the future multi-locale path —
one dump per language). Technologies (with real research-unit counts) come
from `--dump-data` or a small extension of the mod's dump, whichever slice is
cleaner. The existing mod + basisu path stays as-is for the `editor/`
artifact. FactorioLab's scripts are the working reference for dump-flag output
layouts (MIT, easy to read) — verify exact layouts on the pinned 2.0.76 at
implementation time rather than from memory.

## Slice 1 — FIB + browser artifacts on fbe's Pages

Ships end-to-end without touching fbe's website code or deploy pipeline:
exporter changes + committed `browser/` artifacts land in the fbe repo, its
existing Pages deploy publishes them, this app consumes them.

- [x] **1a — exporter: browser artifact** (fbe repo, `packages/exporter/`,
  landed on `claude/fib-data-hosting-bv87mv`). A per-pack `browser/` output:
  the three dump flags (separate invocations; the icon dump needs a display —
  xvfb works), `catalog.json` derived from `data-raw-dump.json` via typed
  partial serde structs, `icons.webp` + `icons.json` composed from the
  game-rendered icon PNGs (64 px cells, 66 px stride, content-deduped; no
  basisu). 25 unit tests on synthetic fixtures; adversarially reviewed
  (locale-refactor byte-identity confirmed, atomic manifest write, dump runs
  disable the injected editor mod — its placeholder icon paths crash
  `--dump-icon-sprites`, confirmed on a real run); `--skip-browser` keeps a
  headless editor-only path. Real-run findings worth remembering: the locale
  dump **does** carry a `descriptions` map (the `.cfg` resolver stays as
  fallback), and Factorio serializes empty Lua arrays as `{}` (handled by a
  tolerant deserializer that still fails loudly on non-empty surprises).
- [x] **1b — publish via fbe's existing data deploy.** All three packs'
  `browser/` tiers are generated from real runs and committed (~18 MB total:
  vanilla 210 items / 212 recipes / 196 techs; space-age 297 / 340 / 275,
  with Space Age's 319 hidden recycling recipes correctly excluded and
  scrap-recycling correctly kept; **space-exploration 749 items / 1004
  recipes / 579 techs on SE 0.7.56 + Factorio 2.0.76** — the version basis
  this app wanted, with real research-unit counts and 141 item
  descriptions). Notably absent from the fbe SE catalog: the FactorioLab
  sxp quirks — 0 dummy pseudo-items (real hidden flags), 1 steam fluid (no
  temperature-variant items), so several of this app's sxp mitigations
  won't need porting to the new source. `packs.json` carries the additive
  fields. Serving URL goes live when the fbe branch merges (Pages deploys
  on master). This app's "never commit game data" constraint is unchanged —
  nothing landed here.
- [ ] **1c — FIB: neutral model + second adapter.** Refactor `PackData` to
  consume a small neutral internal model; the FactorioLab adapter
  (`factoriolab.ts`) and a new `PackSource.kind: "fbe"` adapter both map into
  it. Descriptions and research counts populate `transfer.ts` fields that
  exist and currently carry `""` — no shape changes (the hard constraint
  holds). Icon CSS generation already measures the sheet and emits
  percentage-based rules; the fbe sheet reuses that path (same cell
  geometry), including the `iconText` overlay mechanism if the catalog
  carries overlays.
- [ ] **1d — pack lineup.** Add fbe-sourced `vanilla-2.0` / `space-age` /
  `space-exploration` entries under **new** synthetic combination ids (ids
  are forever; swapping a pack's data basis under an existing id would
  silently repoint users' scoped state and shared URLs). Relabel the FL sxp
  entry "Space Exploration (1.1, FactorioLab)". Decide whether to delist
  (never re-id) the FL vanilla/space-age duplicates once the fbe-sourced
  ones prove out. Default pack switches to the fbe-sourced vanilla.
- [ ] **1e — verification + docs.** Unit tests for the new adapter mapping
  (synthetic fixture, like the FL one); e2e picks up the new packs (the live
  suite then doubles as a canary for *our* data plane, not just FL's);
  update `static-fork.md`'s roadmap + Bigger picture, add a pointer to this
  plan in fbe's docs/CLAUDE.md (they don't mention this fork yet), and note
  progress on fbe issue #8 (it closes in slice 2).

**Slice-1 non-goals:** multi-locale catalogs (the dump-per-language mechanism
is proven by FactorioLab, but en-only ships first); moving the `editor/`
tiers; any fbe website change.

## Slice 2 — FBE transitions to the data repo (later)

More work and coordination — separate effort, tracked in fbe #8/#29 and a
future fbe-side doc when it starts. Sketch, so slice 1 doesn't paint it into
a corner:

- **The dedicated data repo is created here** (own Pages deploy; flat
  history — squash/orphan on regeneration, pack dumps don't need history).
  Both tiers move to it; this app's `packs.ts` base URLs swap over (a
  config-only change — combination ids untouched); fbe keeps the old
  `data/` URLs serving until this app has redeployed.
- fbe's website consumes the data repo via the already-proven `VITE_DATA_URL`
  seam (`__DATA_URL__` → `DATA_ROOT` in `editor/src/common/globals.ts`); the
  dev flow (exporter serving `:8081`) keeps working.
- `data/output/` leaves the fbe repo (becomes gitignored exporter working
  space); the Pages deploy stops baking ~434 MB into `dist/`; e2e strategy
  needs deciding (pin a data-repo snapshot vs. live fetch).
- **Editor-artifact slimming lands here:** lower sprite resolutions and
  **strip animation frames** the editor never draws (it renders frame 0;
  idle-animation work is paused in fbe #29 precisely pending this hosting
  model). Both shrink the hosted tier substantially and improve the fair-use
  posture — the public default becomes a reduced-fidelity asset set, with
  full-quality packs as a bring-your-own/local-exporter option. That is the
  fbe #29 unpause path.
- fbe's blueprint-library panel (fbe #50 phase 5c) can render DOM icons from
  the `browser/` tier at any point after slice 1 — it doesn't need to wait
  for the editor-tier move.

## Open questions

- **Data repo name** — `factorio-pack-data`? (Working name; decide when
  slice 2 creates it.)
- **Delist or keep** the FL-sourced vanilla/space-age duplicates after 1d.
- **`icons.json` vs. rects embedded in `catalog.json`** — decide when the
  icon-sprite dump layout is in front of us; embedded saves a fetch, separate
  keeps the catalog cacheable across icon-only regens. Minor either way.
- **Catalog compression** — Pages serves gzip; if SE's catalog lands big,
  consider precompressed `.json.gz` or trimming, measured not guessed.
